Why We Built This

RMs walk into meetings under-prepared — every time

Pre-meeting research is manual, fragmented, and time-consuming. Most RMs skip it entirely. The intelligence advantage goes to whoever has a quicker analyst team.

  • 30–60 minutes of manual pre-meeting research per client, per meeting — if done at all
  • Fragmented sources — SEC filings, news, LinkedIn, company sites all in separate silos, never synthesised
  • No freshness tracking — stale data reaches the meeting room with no flag
  • Incumbent blind spot — who is the client's current lead bank is often unknown before the pitch
  • Relationship signals missed — board interlocks and alumni ties that could warm a cold call go undetected
  • No portfolio visibility — no consolidated view of deal triggers, covenant risk, or contact recency across the book
  • Institutional knowledge lost — when an RM transitions, their client context leaves with them

Architecture — LangGraph Pipeline

9-Node Sequential Workflow

Each node is a method on BankingResearchOrchestrator. Nodes emit progress events; the dashboard polls /research/status/{job_id} every 2 seconds.

1
scrape_company_info — WebScraperAgentBeautifulSoup + Claude: extracts sector, headcount, HQ, business model, description from company website
2
fetch_financials — EdgarAgentSEC EDGAR 10-K + 10-Q · ticker normalisation (alias map) · disk cache check first · XBRL parse for 20+ financial fields
3
search_news — NewsAgent + NewsClassifierDuckDuckGo queries · LLM batch-classifies 5 items/call: sentiment, severity, materiality, event type, stock impact
4
generate_products — ProductAgentLLM maps client sector + revenue size to likely WF wholesale product portfolio (Treasury, Credit, Capital Markets, etc.)
5
analyze_industry — IndustryAgentNAICS classification via LLM · DuckDuckGo ReAct loop discovers peer companies with tickers + sector trends
6
fetch_peer_financials — EdgarAgent (per peer)10-K / 10-Q per peer company · disk cache check · foreign ticker skip · concurrent ThreadPoolExecutor
7
fetch_officers — OfficerAgentDDG discovery + 4 deep-profile web searches per officer: background · risk_flags · board_memberships · education · confidence
8
apply_temporal_scoring — TemporalDimensionDecay curves per dimension · prune relevance_score < 0.3 · boost material events · compute freshness summary
9
populate_graph — Neo4j + LLMMERGE all 9 node types + relationships · generate AI executive summary across all 7 dimensions · emit DONE event

Data Engineering

TTL Cache Policy — When Data Is Considered Fresh

All cached data in Neo4j expires after a dimension-specific TTL. The pipeline skips re-fetching when the node is still within its TTL window.

DimensionTTLRationale
Financial data (Neo4j node)30 daysRe-fetch if filing node is older; SEC files quarterly so 30d catches updates
Company info (website scrape)7 daysWebsite copy can change; re-scrape weekly for accuracy
News7 daysNews is time-sensitive; stale news misleads sentiment signals
NAICS classification180 daysIndustry codes rarely change; expensive to re-classify
Peer company list180 daysCompetitive set stable; peer financials re-fetched independently
Officer profiles120 daysExecutive roles shift; C-suite tenures average 2–4 years
Board interlocks180 daysBoard seats change at annual meetings (~every 12 months)
Incumbent bank365 daysCredit agreements are annual; revolving facilities multi-year

Two-layer cache

Layer 1 — Disk: SEC filing files. If the raw filing exists on disk, skip the EDGAR download entirely (ignores TTL). 55+ tickers pre-seeded.

Layer 2 — Neo4j: Parsed financial data. If the Financial node is within TTL, skip re-extraction. Re-parse only when Neo4j TTL expires and a new filing exists on disk.

Prune threshold

Items with relevance_score < 0.3 are pruned from the graph at Node 8 (TemporalDimension). Products use a lower threshold (0.21) since they are harder to re-generate.

Data Engineering

Relevance Scoring Methodology

Every item in the graph carries a relevance_score (0.0 – 1.0) computed by TemporalDimension. Used for freshness display, pruning, and urgency boost.

total_score = (recency_score × 0.6) + (content_score × 0.4)

# Recency — within relevance window (linear decay)
recency = 1.0 − (days_old / window) × 0.7   → 0.3 at window edge

# Recency — beyond window (exponential decay)
recency = 0.3 × (1 − decay_rate) ^ extra_days   → floor 0.0

Feature — Covenant Watch

Financial Ratio Monitoring — Thresholds & Logic

Computed from SEC EDGAR filings (10-K and 10-Q). No manual data entry. Ratios update automatically when new filings are processed.

4.0×
Debt / EBITDA threshold
long_term_debt ÷ ebitda · higher is worse · breach = red
2.5×
Interest Coverage threshold
operating_income ÷ interest_expense · lower is worse · breach = red
0 %
Net Margin threshold
net_income ÷ revenue × 100 · lower is worse · negative = immediate flag
3.0×
Debt / Equity threshold
long_term_debt ÷ stockholders_equity · higher is worse
2.0 %
Return on Assets threshold
net_income ÷ total_assets × 100 · lower is worse

Breach Logic

  • Each ratio returns {value, threshold, higher_is_worse}
  • Breach = value crosses threshold in the worse direction
  • Overall status: All Clear (no breach) / Monitor (1 breach) / Breach Risk (2+ breaches)
  • Computed from most-recent filing period available in Neo4j
  • All available periods returned for trend view

What This Replaces

Previously: Credit analyst downloads 10-K, builds Excel model, manually updates ratios each quarter. With Context Fabric: every Research run auto-computes all 5 ratios from EDGAR. No model, no lag.

Extending Thresholds

Thresholds are constants in api.py — easily configurable per sector or credit policy. Current values are broad market standards. Credit team can override per client or industry.

Key Node Properties

  • Financial: 20+ fields from EDGAR — revenue, net_income, ebitda, long_term_debt, cash, assets, equity, interest_expense, FCF, OCF, D&A + filing_period, filing_type
  • Officer: name, role, background_summary, risk_flags, board_memberships[], education[], confidence, discovered_at
  • Company: ticker, sector, headcount, HQ, freshness_updated_at, incumbent_bank_json (cached)
  • Activity / Deal: type, date, contact_name, notes, next_action, status, amount

Why a Graph (not RDBMS)?

Board interlock traversal, alumni path-finding, peer chain queries — these are multi-hop traversals that require 5+ JOINs in SQL. Cypher expresses them in 2 lines. The graph model also naturally represents sparse, variable-schema data from AI extraction.

Security & Compliance

Current Security Posture + Production Path

Production Readiness Checklist

  • ✓
    API key authentication — X-API-Key header required on all /report endpoints. Configurable via BANKING_API_KEY env.
  • ✓
    CORS origin control — BANKING_CORS_ORIGINS env restricts browser origins. Defaults to localhost only.
  • ✓
    OpenAPI spec — auto-generated at /docs, full endpoint documentation
  • ✓
    No credentials in source — all secrets via .env (gitignored). API keys, Neo4j auth, LLM keys.
  • ✓
    Input validation — FastAPI Pydantic models on all POST endpoints. No raw SQL/Cypher injection vectors.
  • ~
    Multi-user / roles — currently single-user. Add OAuth2 / SAML SSO for production multi-RM deployment.
  • ~
    Audit logging — basic activity log per company. Full API audit trail needs middleware addition.
  • ○
    Data residency / encryption at rest — Neo4j running locally. Enterprise deployment needs Neo4j Aura (cloud, encrypted) or on-prem Neo4j Enterprise.
  • ○
    PII / MNPI controls — no current classification of PII fields. Officer data may constitute PII. Compliance review needed before production rollout.

Data Privacy Considerations

  • Officer profiles sourced from public SEC filings and web — not scraped from private sources
  • No client financial data stored beyond what is in public EDGAR filings
  • RM activity notes are stored — classify as MNPI if they contain deal terms
  • LLM inference: Anthropic API sends data off-premise. Ollama runs locally — zero data leaves the network.

Air-gap Option

Set LLM_PROVIDER=ollama to run entirely locally — no data sent to Anthropic. All other sources (EDGAR cache, Neo4j) are on-premise. Full offline operation possible with pre-cached EDGAR data.

Known Limitations

Current Limitations & Mitigations

Honest assessment of what the system cannot yet do reliably — and the planned path for each.

High
Private companies — no EDGAR dataEDGAR only covers SEC-registered companies. Private clients have no 10-K/10-Q. Mitigation: use manual financial input or Pitchbook/Dun & Bradstreet integration in Phase 3.
Med
LLM extraction errorsFinancial field extraction from XBRL is reliable; however, officer profiles and news classification from Ollama (llama3) are lower quality than Claude Sonnet 4.6. Mitigated by confidence scores and manual override buttons.
Med
DuckDuckGo rate limits~10 req/min. Research on a large company with many officers can hit throttling. Mitigated by request spacing in agents. Fix: add a paid search API (SerpAPI, Bing) as a swap-in.
Med
No real-time financial dataAll financials from SEC filings — last quarterly filing may be 3 months old. No intraday or consensus estimates. Fix: Bloomberg/FactSet API in Phase 3.
Med
Product recommendations are LLM-generatedCurrent product portfolio is AI-estimated from sector/size — not drawn from actual WF transaction data. Confidence varies. Fix: integrate real transaction data in Phase 3.
Low
Single-node Neo4jCurrent setup is single Docker container — no HA or replication. Suitable for prototype. Fix: Neo4j Aura or Enterprise cluster for production.
Low
No scheduled refreshResearch is manually triggered from the dashboard. Data ages according to TTL policy but does not auto-refresh. Fix: Celery + Redis cron jobs in Phase 2.
Low
Foreign / private tickersNon-US or non-SEC tickers are skipped gracefully (no EDGAR data). Peer companies outside the US show zero financials. Noted in UI.
